# Делаем домашнюю страничку для своего проекта из подручных средств

## План

1. Зачем нужна персональная страница для вашего проекта?
2. Делаем простейшую лендинг из `README.md` проекта.
3. А теперь делаем красивую страницу.
4. Где нам разместить эту страницу?
5. Настраиваем деплой на Github Pages.
6. Что ещё можно сделать на GitHub Pages?
7. Когда GitHub Pages становится недостаточно.
8. Выводы.

## Доклад

### Зачем нужна персональная страница для вашего проекта?

Целью этого доклада является показать как можно быстро и просто сделать лендинг для вашего проекта. Не важно,
коммерческий это проект или какое-то Open Source решение. Лендинг – это инструмент показать проект во всей красе,
подсветить его сильные стороны и заинтересовать потенциального пользователя.

Т.к. я являюсь архитектором, который в прошлой жизни был разработчиком, мне нравится программировать и я иногда пишу
какие-то проекты для обучения или коммерческого использования. И в этих проектах часто появлялась задача маскировать
sensitive данные в логах.

Под sensitive данными мы подразумеваем, в том числе персональные данные:

> Федеральный закон № 152-ФЗ «О персональных данных» регулирует обработку персональных данных физических лиц и
> направлен на защиту их прав и свобод, в частности права на неприкосновенность частной жизни.
> Кратко: закон определяет, как можно собирать, хранить, использовать и передавать персональные данные, и вводит
> ограничения для операторов данных (компаний, сервисов, госорганов).
> Запрещено:
> – хранить данные без необходимых мер безопасности;
> – допускать утечки, несанкционированный доступ.

Отсюда следует, что если у вас в логах появляются ФИО + email, то это уже персональные данные, которые вы не можете
отправлять в какую-то облачную систему агрегации логов. Следовательно, нужно эти данные маскировать. Чтобы не
реализовывать эту функциональность одинаково в нескольких проектах, я вынес код в библиотеку и опубликовал на
Github ([log4j2-masking-lib](https://github.com/Romanow/log4j2-masking-lib)).

На мой взгляд реализация получилась неплохой и я задумался над тем, чтобы это решение могли использовать другие.

Вначале проект выглядел вот так:

![Without README.md](images/Without%20README.md.png)

Как вы понимаете, шанс что кто-то захочет использовать это решение равнялся примерно нулю. Нужно было:

* описать решение;
* показать как настраивать и кастомизировать;
* оформить это так, чтобы было приятно с этим работать.

Первое, что нужно сделать – это описать `README.md` в проекте, где рассказать про то как подключать библиотеку,
кастомизировать и описать внутреннее устройство.

Получилось вот так:

Согласитесь, получилось лучше: вы сразу видите как подключить и настроить библиотеку.

через Github Pages (кстати, в Gitverse тоже появились pages) и даже подключить свой домен:

### А теперь делаем красивую страницу

На этом этапе у нас уже есть базовый лендинг вашего проекта. Теперь давайте сделаем его красивым.

Здесь давайте немного разомнемся: кто только недавно вошел в IT?

Сколько бы времени делали небольшой лендинг для этого проекта? День / два?

Давайте это сделаем за 10 минут?

```text
Мне нужно сделать лендинг для старицы проекта log4j2-masking-lib описывающий проект. Стиль формальный, описание возьми из README.md.

Структура страницы:
– Заголовок: Open Source, название проекта, описание, ссылка на GitHub и на Latest Release, а так же список бейджей из README.md.
– Тело сайта раздели на строки по 1 или 2 колонки:
    – На первой строке "Назначение".
    – Дальше идет две колонки: "Подключение" и "Конфигурация правил".
    – Далее строка "Интеграция с Log4j2".
    – Потом "Правила маскирования" в виде таблицы (добавь примеров из тестов).
– Footer – ссылка на автора (Романов Алексей: https://github.com/Romanow, https://romanow-alex.ru/) и на лицензию.

CSS выноси в отдельный файл. Для CSS можешь использовать библиотеку MUI, либо сделать свой CSS под Material UI.
Результат положи в папку landing.
```

Теперь для деплоя страницы нужно либо перенести этот код в отдельную ветку (create orphan branch
`git checkout --orphan gh-pages`) и настроить _Deploy from a branch_, либо выбрать GitHub Actions:

```yaml
name: Deploy landing
on:
    push:
        branches:
            - master
        paths:
            - "landing/**"

jobs:
    deploy:
        environment:
            name: github-pages
            url: ${{ steps.deployment.outputs.page_url }}
        runs-on: ubuntu-latest
        permissions:
            id-token: write
            contents: read
            pages: write
        steps:
            -   name: Checkout
                uses: actions/checkout@v4

            -   name: Setup Pages
                uses: actions/configure-pages@v5

            -   name: Upload landing artifact
                uses: actions/upload-pages-artifact@v3
                with:
                    path: ./landing

            -   name: Deploy to GitHub Pages
                id: deployment
                uses: actions/deploy-pages@v4

```

### Что ещё можно сделать на GitHub Pages?

К слову, разметка Markdown отлично подходит для ведения технической документации. На GitHub поддерживает дополнительный
синтаксис:

* Checkbox;
* Callouts (блоки уведомлений);
* Mermaid диаграммы;
* HTML внутри Markdown (есть whitelist html тегов, которые разрешены);

Это может показаться странным, ведь есть ~~Confluence~~ множество других "удобных" инструментов для ведения
документации? (а помните как мы раньше хейтили Confluence?)

Но Markdown в связке с git дает удобное версионирование, а так же простой поиск. Сейчас к этому можно легко подключить
ИИ агент, ведь ему по сути нужен будет доступ к текстовым файлам, которые хранятся локально, а не к огромной и
запутанной базе данных Confluence.

Я сам к этому решению пришел не сам: в 2020 мы с друзьями готовили курс лекций для одной компании и нам надо было вести
общий документ с лекциями. Я предложил Google Docs, но достаточно быстро стало понятно, что никаких особых функций Word
нам не нужно, а работать параллельно крайне неудобно. Так мы создали репозиторий на GitHub и дальше работали как
привыкли с PR, ревью и конфликтами слияния.

Сейчас это называется DocOps.

Отлично, у нас документация, теперь мы хотим ее оформить красиво, например как в проекте Spring:

![Spring documentation](images/Spring%20documentation.png)

Инструментов для оформления документации в таком виде много, я обычно использую [Docusaurus](https://docusaurus.io/) –
Open Source инструмент, для создания сайтов документации из Markdown-файлов с использованием React.

Мы пишем документацию как код, а Docusaurus трансформирует Markdown файлы в полноценный сайт с навигацией, поиском и UI:

* **Навигация**: sidebar, автоматическая структура;
* **UI**: готовый интерфейс, адаптированный к мобильным устройствам;
* **Поиск**: встроенный поиск по документации;
* **Версионирование**: можно хранить версии документации (v1, v2, ...);
* **Markdown + React**: можно писать Markdown и при необходимости встраивать React-компоненты.

Мы складываем документацию в папку `/docs`:

```text
docs/
├── intro.md
├── docker-architecture.md
├── docker-cli-commands.md
├── dockerfile.md
├── docker-compose.md
├── container-networking.md
├── volumes.md
├── orchestration-prerequisites.md
└── docker-swarm.md
```

Описываем структуру в файле [sidebars.js](https://github.com/IT-Enduro/docker-lecture/blob/master/sidebars.js):

```javascript
module.exports = {
    lectureSidebar: [
        'intro',
        'orchestration-prerequisites',
        'docker-architecture',
        'docker-cli-commands',
        'dockerfile',
        'container-networking',
        'volumes',
        'docker-compose',
        'docker-swarm'
    ]
};
```

И описываем структуру
в [docusaurus.config.js](https://github.com/IT-Enduro/docker-lecture/blob/master/docusaurus.config.js):

```javascript
const config = {
    title: 'Интенсив по Docker',
    favicon: 'images/logo.svg',
    url: 'https://it-enduro.com',
    baseUrl: '/docker-lecture/',
    organizationName: 'it-enduro',
    projectName: 'docker-lecture',
    presets: [
        [
            'classic',
            {
                docs: {
                    sidebarPath: require.resolve('./sidebars.js'),
                    routeBasePath: '/'
                }
            }
        ]
    ],
    themeConfig: {}
};

module.exports = config;

```

После этого запускаем `npm run start` и наша документация доступна по адресу [localhost:3000](http://localhost:3000).

Теперь мы можем этот сайт опубликовать на GitHub Pages через Actions:

> [!WARNING]
> Для бесплатной версии GitHub ваша страница на GitHub Pages может быть только публичной.
> Можно реализовать через связку User → Cloudflare Access → GitHub Pages, но страница все равно будет доступна по прямой
> ссылке [it-enduro.github.io/docker-lectures](http://it-enduro.github.io/docker-lectures/).

### Когда GitHub Pages становится недостаточно

Если вам требуется интеграция с платежными формами, авторизация, хранение данных на backend, то статической страницы
будет уже недостаточно.
